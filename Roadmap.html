Step by Step Implementation

Each row tells the full story of what happens at that step — tracing every technology from the Blazor UI through Foundry, Azure AI, SQL Server, and SharePoint.

Step What Happens — Full Stack Narrative Stack Used
1 Upload File
The Bid Manager opens the Blazor web app and logs in using their credentials. A JWT token is issued with their role (Bid Manager), which unlocks the Upload File screen. They select the document and click Upload — Blazor sends an HTTP POST to the Foundry Agent Service, which immediately creates a new durable agent session (a "thread") for this proposal. A unique proposal ID is returned and stored. Everything from this point forward runs inside that single persistent thread — no separate flows, no polling loops.
Blazor App Foundry Agent Service Durable Session Created
Blazor · Foundry
2 File & Record Storage
The Foundry agent, now running inside its durable session, calls two MCP tools in sequence. First, the Azure Blob Storage MCP server receives the document binary, stores it securely, and returns a blob URL. Second, the SQL Server MCP server inserts a new Proposal record into the database with the blob URL, a timestamp, and the Bid Manager's user ID. Both the blob URL and the SQL record ID are stored directly in the agent's persistent thread memory — no manual expression mapping, no credential juggling. Authentication is handled entirely by Foundry.
Foundry Agent MCP: Azure Blob MCP: SQL Server Thread Memory Updated
Blob · SQL Server · Foundry
3 Uploaded File Parsing
The agent calls the Azure Document Intelligence MCP tool, passing the blob URL from thread memory. Document Intelligence reads the Uploaded document and extracts structured data — client name, project scope, requirements list, budget ceiling, submission deadline, and compliance criteria . Foundry handles the async wait internally; from the agent's perspective this is a single synchronous tool call. The extracted fields are written back into the agent's thread memory as a clean JSON object, ready to be used in every downstream step without re-processing the document.
Foundry Agent MCP: Azure Doc Intelligence Async Polling — Internal Extracted Fields → Thread Memory
Doc Intelligence · Foundry
4 Knowledge Retrieval
Using the extracted requirements from thread memory, the agent invokes Foundry IQ — a single call that fans out across all four Azure AI Search semantic indexes simultaneously: past projects, team profiles, reference documents, and pricing templates. Foundry IQ applies cross-index semantic reranking to surface the most relevant results. It then runs iterative reflection — if the initial results are insufficient, it automatically re-queries with a refined strategy before returning a single clean, ranked context package to the agent. No parallel code. No manual result merging.
Foundry IQ Azure AI Search — 4 Indexes Semantic Reranking Iterative Reflection
Foundry IQ · AI Search
5 Context Assembly
Foundry IQ delivers a deduplicated, confidence-filtered context package back to the agent thread — the top matching past projects, best-fit team members, relevant reference document sections, and applicable pricing templates. Each result carries provenance (which index it came from and its relevance score). The agent does not need to know how retrieval worked — it simply receives a ready-to-use context window. This assembled context, combined with the Uploaded File extract already in thread memory, becomes the full input for the generation step.
Foundry IQ — Context Package Deduplication + Confidence Filter Provenance Tracked Ready for Generation
Foundry IQ
6 AI Section Generation
The agent passes the full context — Uploaded File requirements + retrieved knowledge — to GPT-4o via the Foundry model catalogue. The model is configured with structured output mode, so Foundry enforces a strict JSON schema: five proposal sections (Executive Summary, Technical Approach, Team Profiles, Pricing Narrative, Project Timeline), each with inline reference. GPT-4o writes all five sections in one generation pass, using the company's voice and the Uploaded File's specific language. The model is swappable — switching to Anthropic Claude or Meta Llama requires only a config change, no code deployment. All five section objects are stored in the agent's thread memory.
Foundry Model Catalogue Azure OpenAI GPT-4o Structured Output Mode 5 Sections → Thread Memory
GPT-4o · Foundry
7 Save & Display
The agent calls the SQL Server MCP tool to write Proposal Section rows into the database, each linked to the Proposal record by ID. It then updates the Proposal record's status to Draft Ready. The Blazor app is listening via a SignalR hub — the moment the SQL write completes, the Bid Manager's workbench page refreshes in real time and displays all five sections. Because the full draft also lives in Foundry's persistent thread memory, the Bid Manager can open the AI Workspace panel in Blazor, type a question — "why did you select this team?" — and the agent answers instantly from thread context, without any new search query. Blazor also triggers in-app notifications for all assigned reviewers.
MCP: SQL Server — 5 Section Rows Blazor SignalR — Live Refresh Thread Memory — Instant Q&A Reviewer In-App Notifications
SQL Server · Blazor · SignalR
8 Review & Approval
Each reviewer receives an in-app notification and logs into the Blazor app with their credentials. Their JWT token contains role claims that determine which sections they can see — the Tech Lead sees only Technical Approach, the Delivery Manager sees only Project Timeline, the Finance Lead sees only Pricing Narrative, and the Executive reviews the Executive Summary. Within their section view, each reviewer can annotate, request a change, or click Approve. They can also open the AI Workspace and ask the Foundry agent questions about the section — getting grounded answers from the proposal's thread memory. Once all five sections are approved by their respective reviewers, the SQL Server record is automatically updated to Approved status and the pipeline advances to delivery.
Blazor — Role-Gated Section View JWT Role Claims Foundry Thread — Inline AI Q&A SQL Server — Status: Approved
Blazor · Foundry · SQL Server
9 Delivery & Audit
On approval, the Foundry agent retrieves the final proposal sections from thread memory. It fetches the approved Word template from SharePoint using Microsoft Graph. The agent maps all sections into the template using structured content controls to preserve formatting. A fully formatted DOCX is generated within the agent workflow. The document is stored in Azure Blob Storage for instant access. The Blazor app receives a SignalR update and shows a “Download Final Proposal” button. The same document is optionally uploaded to SharePoint for archiving and external sharing. SQL Server is updated with Submitted status, timestamp, and document links. The Foundry Control Plane captures the complete execution trace. Built-in evaluators score the proposal for quality, coherence, and compliance.
MCP: SharePoint MCP: SQL Server — Status: Submitted Blob URL: Direct Access in Blazor Microsoft Graph API: Optional SharePoint Upload Blazor — Delivery Confirmation Foundry Control Plane — Full Trace Built-in Evaluators — Quality Score
SharePoint · SQL · Blazor · Control Plane · Azure Blob

Step-by-Step Implementation · All 9 steps mapped to Foundry
The 9-Step Data Flow — Built on Foundry
1
Upload · Blazor App · entry point
Bid manager uploads Uploaded File via Blazor — triggers the Foundry Agent session
The UI entry point is the Blazor workbench — a bid manager uploads the Uploaded File directly from the Blazor web app. The upload fires an HTTP request to the Foundry Agent Service's durable session endpoint. Foundry creates a persistent "thread" for this proposal — a memory space that survives across all 9 steps and can be queried by the AI assistant later.
Foundry approach
HTTP POST to Foundry Agent Service durable session endpoint → agent thread created → all 9 steps run as agent tool calls within the same persistent session
Blazor App — Document Upload UI Foundry Agent Service — durable session start Thread created — persists across all 9 steps
2
Storage + Record creation · MCP tools replace manual HTTP
Agent calls MCP tools to upload file and create the SQL Server record
The agent calls two registered MCP tools: the Azure Blob MCP server and the SQL Server MCP server. The agent passes the file binary and the required fields; the MCP tools handle authentication, the REST calls, and error handling. No custom HTTP code to maintain.
Foundry component — MCP tool calls
MCP Server: Azure Blob + MCP Server: SQL Server
The Foundry agent calls registered MCP tools as part of its reasoning step. Each MCP server exposes structured operations: the Blob server exposes UploadFile(blobName, data) → returns URL. The SQL Server MCP server exposes CreateRecord(tableName, fields) → returns record ID. The agent stores both results in its persistent thread memory for use in later steps.
MCP: Azure Blob — UploadFile tool MCP: SQL Server — CreateRecord tool blobUrl stored in agent thread memory proposalId stored in agent thread memory
3
Document parsing
Foundry calls the Document Intelligence MCP tool
Foundry's MCP tool abstraction handles asynchronous tool calls internally. The agent calls the Document Intelligence MCP tool, Foundry manages the wait-and-poll cycle, and the agent receives the structured extraction result as a clean tool response. To the agent, it's a synchronous call.
Foundry — MCP tool call
Agent calls: AnalyseDocument(blobUrl) → Foundry handles async polling internally → agent receives: { clientName, scope, requirements, budget, deadline, compliance }
MCP: Azure Document Intelligence Async polling handled by Foundry internally Structured extraction result → agent memory Zero polling loop code in the agent definition
4
Knowledge retrieval
Foundry IQ fans out across all four knowledge indexes with iterative reflection
Foundry IQ (Foundry's next-generation RAG engine), the agent simply asks a question and Foundry IQ handles the multi-index fan-out automatically. Foundry IQ also applies iterative reflection — after the initial retrieval, it evaluates the results and retrieves again if the quality isn't sufficient.
Foundry component — Foundry IQ
Foundry IQ — multi-source retrieval with iterative reflection
Foundry IQ is configured with all four Azure AI Search indexes as knowledge sources. When the agent invokes retrieval, Foundry IQ fans the query out across all four indexes simultaneously, applies semantic reranking across the combined result set, and then uses iterative reflection — evaluating whether the retrieved context is sufficient to answer the question and re-querying if not. The agent receives a single, clean, ranked context package. No parallel scope code. No custom result assembly. No confidence threshold filtering in application code.
past-projects-index team-profiles-index ref-documents-index pricing-templates-index Iterative reflection — auto-retry if quality low Cross-index semantic reranking
→ Four parallel HTTP calls replaced by one Foundry IQ invocation — with higher quality retrieval through iterative reflection
5
Context assembly · Foundry IQ handles ranking and deduplication
Ranked, deduplicated context assembled automatically
By the time the agent's model is invoked, it receives a single clean context window — already ranked, deduplicated, and filtered — sourced from all four indexes. The agent doesn't need to know which index each piece came from; Foundry IQ handles provenance and citation automatically.
Top 5 similar past projects — ranked by semantic similarity Top 8 matched available team members Relevant reference document sections Applicable pricing templates Confidence threshold filtering applied by Foundry IQ Provenance tracked — citations auto-generated
→ Context assembly is zero-code in Foundry — Foundry IQ produces a clean, grounded context package automatically
6
AI generation · Model catalogue + structured output mode
Agent generates all 5 sections — structured output enforced, model switchable
Foundry, the model is selected from the model catalogue — GPT-4o today, Anthropic Claude or Meta Llama tomorrow, without changing any application code. Structured output mode is configured in the agent definition: Foundry enforces the JSON schema at the API level, so the agent always returns the exact structure expected. No manual parsing. No schema drift.
Foundry component — Model catalogue + structured output
Model catalogue selection + structured output mode
The agent's model is defined in the Foundry agent configuration as a pointer to the model catalogue — not a hardcoded endpoint URL. Switching models is a single config change. Structured output mode is declared in the agent definition: the expected JSON schema (executiveSummary, technicalApproach, teamProfiles, pricingNarrative, projectTimeline — each with citations[]) is validated by Foundry before the response is returned to the agent. The agent writes the five section objects directly to its thread memory, from where the SQL Server MCP tool reads them in Step 7.

Persistence + display · MCP: SQL Server + Foundry thread state
Agent writes sections to SQL Server via MCP — thread state enables instant AI Q&A in Blazor
The agent calls the SQL Server MCP tool to write the five Proposal Section records. Blazor's SignalR hub picks up the changes and renders the sections in the workbench in real time. The sections also live in the Foundry agent's persistent thread memory — the moment the draft is ready, the bid manager or any reviewer can open the AI Workspace in Blazor and ask questions about the specific proposal, and the agent answers immediately from thread context.

8
Review & Approval · Blazor Role-Based Workflow · Section-wise sign-off
Reviewers log in to Blazor with their role — each approves their assigned section
Once the AI draft is ready, Blazor notifies all assigned reviewers via in-app alerts. Each reviewer logs in with their role-scoped credentials and sees only the sections they are responsible for. A Technical Lead reviews the Technical Approach, a Delivery Manager reviews the Project Timeline, a Finance Lead reviews the Pricing Narrative — all within the same Blazor dashboard, without email chains or shared documents.
Blazor Role-Based Access — Review & Approval
Role-wise login · Section-scoped review · In-app approval workflow
JWT token role claims control what each user can see and do inside Blazor. The Bid Manager role has upload and submission rights. Each Reviewer role (Technical, Delivery, Finance, Executive) is scoped to specific proposal sections. Reviewers can annotate, request changes, or approve directly in the Blazor interface. The Foundry agent thread holds the full draft — reviewers can ask the AI questions about any section inline. Once all sections are approved, the proposal is automatically promoted to the delivery step.
Role: Bid Manager — Upload & Submit Role: Tech Lead — Technical Approach review Role: Delivery Manager — Timeline review Role: Finance Lead — Pricing review Role: Executive — Executive Summary sign-off Role: Admin — User & role management JWT role claims — section-scoped visibility In-app approval — no email required
→ All review and approval happens inside Blazor — role-gated, section-wise, with inline AI Q&A on the draft
9
Delivery · MCP tools + Foundry Control Plane trace
Agent packages and delivers the proposal — every action traced in Control Plane
On approval, the agent generates the final proposal using the SharePoint template and structured content mapping. It then calls MCP tools in sequence to store, update, and deliver the document. The final DOCX is saved in Azure Blob for instant access in Blazor. The same file is uploaded to SharePoint for archiving and external sharing. SQL Server is updated with Submitted status and document links. Blazor notifies the Bid Manager via an in-app alert with download access. All actions — tool calls, model outputs, and updates — are recorded in the Control Plane. The entire process is fully traceable, replayable, and auditable.
Foundry component — Control Plane full trace
Foundry Control Plane — complete agent observability
Every step of the proposal lifecycle is captured end-to-end. This includes document extraction results, retrieval queries, and model inputs/outputs. Database updates and delivery actions are also logged automatically. Users can open any proposal and replay its full execution history. Built-in evaluators score proposals for groundedness, coherence, and safety. Low-quality or failed outputs are flagged for review. Continuous evaluation ensures consistent quality across all proposals.
