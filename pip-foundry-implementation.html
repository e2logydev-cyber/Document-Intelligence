Step-by-Step Implementation · All 9 steps mapped to Foundry
The 9-Step Data Flow — Rebuilt on Foundry

Each step below shows what the original implementation does, what Foundry replaces it with, and why the Foundry approach is simpler, more observable, and more flexible. Steps are colour-coded by the primary Foundry component involved.

1
Upload · Blazor / Power Apps · unchanged entry point
Bid manager uploads RFP — triggers the Foundry Agent session
What changes with Foundry
The UI entry point is identical — a bid manager still uploads via the Blazor workbench or the Power Apps mobile app. What changes is what happens after the upload. Instead of triggering a Power Automate flow via SharePoint, the upload fires an HTTP request to the Foundry Agent Service's durable session endpoint. Foundry creates a persistent "thread" for this proposal — a memory space that survives across all 9 steps and can be queried by the AI assistant later.
Original approach
SharePoint file trigger OR Power Apps HTTP → fires a Power Automate cloud flow that runs linearly from step to step
With Foundry
HTTP POST to Foundry Agent Service durable session endpoint → agent thread created → all 9 steps run as agent tool calls within the same persistent session
Blazor or Power Apps — unchanged UI Foundry Agent Service — durable session start Thread created — persists across all 9 steps
→ UI unchanged — all complexity moves into the Foundry agent definition, not the application code
2
Storage + Record creation · MCP tools replace manual HTTP
Agent calls MCP tools to upload file and create the Dataverse record
What changes with Foundry
In the original implementation, Power Automate makes two manual HTTP calls — one to the Azure Blob REST API to upload the file, and one via the Dataverse connector to create the Proposal record. With Foundry, the agent calls two registered MCP tools: the Azure Blob MCP server and the Dataverse MCP server. The agent passes the file binary and the required fields; the MCP tools handle authentication, the REST calls, and error handling. No custom HTTP code to maintain.
Foundry component — MCP tool calls
MCP Server: Azure Blob + MCP Server: Dataverse
The Foundry agent calls registered MCP tools as part of its reasoning step. Each MCP server exposes structured operations: the Blob server exposes UploadFile(blobName, data) → returns URL. The Dataverse server exposes CreateRecord(tableName, fields) → returns record ID. The agent stores both results in its persistent thread memory for use in later steps.
MCP: Azure Blob — UploadFile tool MCP: Dataverse — CreateRecord tool blobUrl stored in agent thread memory proposalId stored in agent thread memory
→ Two MCP tool calls replace two Power Automate HTTP actions — agent memory holds the results for downstream steps
3
Document parsing · MCP tool replaces Do Until polling loop
Foundry calls the Document Intelligence MCP tool — no polling loop required
What changes with Foundry
The most tedious part of the original pipeline — the Do Until polling loop that checks Document Intelligence's status every 5 seconds until it says "succeeded" — disappears entirely. Foundry's MCP tool abstraction handles asynchronous tool calls internally. The agent calls the Document Intelligence MCP tool, Foundry manages the wait-and-poll cycle, and the agent receives the structured extraction result as a clean tool response. To the agent, it's a synchronous call.
Original — Do Until loop
HTTP POST → get Operation-Location → loop: GET every 5s → check status field → wait → repeat until "succeeded" → parse JSON → extract 6 fields manually
With Foundry — MCP tool call
Agent calls: AnalyseDocument(blobUrl) → Foundry handles async polling internally → agent receives: { clientName, scope, requirements, budget, deadline, compliance }
MCP: Azure Document Intelligence Async polling handled by Foundry internally Structured extraction result → agent memory Zero polling loop code in the agent definition
→ The Do Until loop is gone — Foundry's async MCP call abstraction eliminates the most complex part of the original flow
4
Knowledge retrieval · Foundry IQ replaces 4 parallel SDK calls
Foundry IQ fans out across all four knowledge indexes with iterative reflection
What changes with Foundry
In the original implementation, Power Automate's "Run in Parallel" scope fires four separate Azure AI Search HTTP calls at the same time — each manually constructed with the right endpoint, query vector, and field mappings. With Foundry IQ (Foundry's next-generation RAG engine), the agent simply asks a question and Foundry IQ handles the multi-index fan-out automatically. Foundry IQ also applies iterative reflection — after the initial retrieval, it evaluates the results and retrieves again if the quality isn't sufficient. This isn't possible with a static Power Automate parallel scope.
Foundry component — Foundry IQ
Foundry IQ — multi-source retrieval with iterative reflection
Foundry IQ is configured with all four Azure AI Search indexes as knowledge sources. When the agent invokes retrieval, Foundry IQ fans the query out across all four indexes simultaneously, applies semantic reranking across the combined result set, and then uses iterative reflection — evaluating whether the retrieved context is sufficient to answer the question and re-querying if not. The agent receives a single, clean, ranked context package. No parallel scope code. No custom result assembly. No confidence threshold filtering in application code.
past-projects-index team-profiles-index ref-documents-index pricing-templates-index Iterative reflection — auto-retry if quality low Cross-index semantic reranking
→ Four parallel HTTP calls replaced by one Foundry IQ invocation — with higher quality retrieval through iterative reflection
5
Context assembly · Foundry IQ handles ranking and deduplication
Ranked, deduplicated context assembled automatically — no manual result merging
What changes with Foundry
In the original pipeline, the results from four separate searches had to be manually assembled inside the Power Automate flow — parsing each search response, filtering by confidence score, ranking within each result set, and then combining into a single context string. Foundry IQ does all of this automatically. By the time the agent's model is invoked, it receives a single clean context window — already ranked, deduplicated, and filtered — sourced from all four indexes. The agent doesn't need to know which index each piece came from; Foundry IQ handles provenance and citation automatically.
Top 5 similar past projects — ranked by semantic similarity Top 8 matched available team members Relevant reference document sections Applicable pricing templates Confidence threshold filtering applied by Foundry IQ Provenance tracked — citations auto-generated
→ Context assembly is zero-code in Foundry — Foundry IQ produces a clean, grounded context package automatically
6
AI generation · Model catalogue + structured output mode
Agent generates all 5 sections — structured output enforced, model switchable
What changes with Foundry
In the original implementation, the model is hardcoded as GPT-4o, the JSON response is parsed manually in Power Automate expressions, and if the schema changes the flow breaks. With Foundry, the model is selected from the model catalogue — GPT-4o today, Anthropic Claude or Meta Llama tomorrow, without changing any application code. Structured output mode is configured in the agent definition: Foundry enforces the JSON schema at the API level, so the agent always returns the exact structure expected. No manual parsing. No schema drift.
Foundry component — Model catalogue + structured output
Model catalogue selection + structured output mode
The agent's model is defined in the Foundry agent configuration as a pointer to the model catalogue — not a hardcoded endpoint URL. Switching models is a single config change. Structured output mode is declared in the agent definition: the expected JSON schema (executiveSummary, technicalApproach, teamProfiles, pricingNarrative, projectTimeline — each with citations[]) is validated by Foundry before the response is returned to the agent. The agent writes the five section objects directly to its thread memory, from where the Dataverse MCP tool reads them in Step 7.

7
Persistence + display · MCP: Dataverse + Foundry thread state
Agent writes sections to Dataverse via MCP — thread state enables instant AI Q&A
What changes with Foundry
The Dataverse write works the same way — the five ProposalSection records are created using the Dataverse MCP tool. Blazor's SignalR hub picks up the changes and renders the sections in the workbench. The important addition is that the sections also live in the Foundry agent's persistent thread memory. This means the moment the draft is ready, the bid manager can open the AI Workspace in Blazor and ask questions about the specific proposal — "why did you choose this team member?", "what past project supports the technical approach?" — and the agent answers immediately from thread context, without a separate search query.
MCP: Dataverse — CreateRecord × 5 ProposalSection MCP: Dataverse — UpdateRecord Proposal status Agent thread memory — full draft + citations retained Blazor SignalR — real-time section rendering Instant proposal Q&A — no re-search needed
→ Thread memory enables zero-latency Q&A on the draft — the agent already knows everything about this proposal
8
Review & approval · MCP: Teams + agent-generated Adaptive Card
Agent sends Adaptive Card to Teams — card content generated from thread memory
What changes with Foundry
In the original implementation, the Adaptive Card JSON is manually constructed in Power Automate using concatenated expressions — a brittle, hard-to-maintain approach. With Foundry, the agent calls the Teams MCP tool and provides the card content as natural language instructions. Foundry generates the Adaptive Card JSON automatically from the agent's structured output. Additionally, reviewers can respond to the Adaptive Card by typing in Teams, and the Foundry agent reads their response — enabling a conversational review flow rather than just button clicks.
Foundry component — MCP: Microsoft Teams
Teams MCP server — Adaptive Card generation + conversational review
The Teams MCP server exposes SendAdaptiveCard(channel, content) as a tool. The agent generates the card content from its thread memory — the right sections for the right reviewer role are selected automatically from the structured proposal data. Reviewers' responses (approve, request changes, comments) are read back by the agent via the Teams MCP ListMessages tool. This enables a conversational approval loop — the reviewer can ask the agent clarifying questions directly in Teams and receive grounded answers from the proposal context.
MCP: Teams — SendAdaptiveCard MCP: Teams — ListMessages (read responses) Role-scoped card content — from agent thread Conversational review — natural language in Teams
→ Card content from agent thread — no manual JSON construction, conversational review enabled
9
Delivery · MCP tools + Foundry Control Plane trace
Agent packages and delivers the proposal — every action traced in Control Plane
What changes with Foundry
On approval, the agent calls three MCP tools in sequence: the SharePoint MCP server (exports the proposal as PDF and DOCX, uploads to the archive library), the Dataverse MCP server (updates CRM status to Submitted), and the Teams MCP server (notifies the bid manager). Power BI refreshes as before via Dataverse change notifications. The critical addition is that every single action the agent took across all nine steps — every MCP tool call, every model invocation, every retrieval query — is recorded in the Foundry Control Plane. You can replay, audit, and evaluate any proposal that has ever been processed.
Foundry component — Control Plane full trace
Foundry Control Plane — complete agent observability
Every step the agent took for this proposal is captured: the Document Intelligence tool call result, the Foundry IQ retrieval queries and their ranked results, the model input context and output, the Dataverse write confirmations, and the Teams delivery receipts. You can open any completed proposal in the Foundry portal and replay its entire processing history. Built-in evaluators score every proposal for groundedness (are the citations real?), coherence (does the proposal read well?), and safety (does it meet content policy?). Failed or low-quality proposals are flagged automatically.

Foundry implementation complete

Benefits of the Foundry approach
What You Gain with Foundry

Beyond replacing manual wiring, Foundry unlocks capabilities that simply aren't possible with the original Power Automate + SDK approach.

Foundry Control Plane
Full observability — every agent action traced
Every tool call, retrieval query, model invocation, and output across all nine steps is recorded in the Foundry Control Plane. You can audit any proposal, replay its processing history, and see exactly why the AI made each decision. This is impossible in a Power Automate flow without building custom logging infrastructure.
Model catalogue
Switch models without touching application code
The model is a pointer in the agent definition, not a hardcoded endpoint URL. Moving from GPT-4o to Anthropic Claude or Meta Llama is a single configuration change — no code deployments, no flow updates, no endpoint string changes across multiple services. You can A/B test models on real proposals and measure quality differences in the Control Plane.
Foundry IQ
Iterative reflection improves retrieval quality
Foundry IQ doesn't just run one retrieval pass. After the initial search across all four indexes, it evaluates whether the results are sufficient and re-queries if not. This iterative reflection produces better context for the model — more relevant past projects, better-matched team members — without any additional code. The original parallel Power Automate scope retrieves once and stops.
Persistent thread memory
Zero-latency Q&A on any processed proposal
Because the Foundry agent maintains durable thread state across all nine steps, the full proposal context — RFP extract, search results, generated sections, citations — lives in the agent's memory. The bid manager can ask questions about the proposal in Blazor or Teams and receive instant, grounded answers without triggering a new search cycle.
Continuous evaluations
Catch quality regressions before they reach users
The Foundry Control Plane's built-in evaluators run automatically on every processed proposal — scoring for groundedness (are citations real?), coherence (does the proposal read well?), and safety. You can define custom domain-specific evaluators and run them continuously. If a change to the knowledge base or model version degrades proposal quality, the evaluators flag it before a bid manager notices.
MCP tool ecosystem
Connect any service without custom HTTP code
Any MCP server in the Foundry ecosystem — today Azure Blob, Dataverse, Document Intelligence, SharePoint, Teams — and tomorrow Salesforce, Jira, Slack, or any custom internal service — becomes available to the agent as a typed tool call with authentication, retry, and error handling managed by Foundry. No custom HTTP code. No credential management in flow expressions.
